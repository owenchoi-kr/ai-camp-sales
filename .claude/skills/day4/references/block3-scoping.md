# Block 3: Scoping — 2주차 프로젝트 스코핑 (오늘의 산출물)

> 오늘 하루의 목적지. 2주차에 만들 프로젝트를 고르고, Claude가 인터뷰하며 스코핑 1장을 채워 **`my-project.md`로 저장**한다.
> 이 블록만 STOP PROTOCOL을 조금 다르게 쓴다: Phase A는 메뉴 소개 + 인터뷰 예고 후 Stop, Phase B에서 대화형 인터뷰 + Write.

## EXPLAIN (Phase A)

> **Phase A 시작 시 반드시 아래 형태로 출력한다:**
> ```
> 📖 공식 문서: https://code.claude.com/docs/ko/skills
> ```

### 2주차 — 내 실제 딜에 하나 적용

1주차엔 실습용으로 만들었다. 2주차엔 **내 실제 파이프라인 1개**에 적용하는 프로젝트를 만든다. 기준은 딱 하나: **내 실제 딜/루틴에서 매주 반복되는 시간 낭비를 없앨 것.** 새로운 걸 만드는 게 아니라, Block 2에서 짠 루틴의 빈 스텝을 채우는 것에 가깝다.

### 메뉴 소개

Claude는 **`docs/project-menu.md`를 Read**해서 메뉴 6개(표)를 그대로 보여준다. 요약하지 말고 표를 그대로 제시한다. 6개 요지:

1. 내 분기 딜 현황 스킬 · 2. 내 옵티 위생 점검 스킬 · 3. 세일즈 Q&A 스킬(팀 확산 임팩트 최대) · 4. 이벤트 초청 트래킹 스킬 · 5. 온보딩 핸드오프 문서 스킬 · 6. Day 2~3 산출물 심화

> 메뉴에 없어도 된다 — **자기 고통점이 더 구체적이면 그걸로** 정의한다. Block 1의 검증 경험이 힌트를 준다("내가 매번 아쉬웠던 게 뭐였지?").

이제 인터뷰할 **4가지**를 예고한다 (docs/project-menu.md의 스코핑 1장 템플릿과 동일):

1. **없애려는 반복 작업** — 무엇을, 얼마나 자주, 지금 걸리는 시간
2. **완성 기준** — 이걸 보면 "됐다" (실제 딜 1개 적용 포함)
3. **범위 밖** — 이번엔 안 하는 것 (자동 발송·팀 일반화 등)
4. **쇼케이스 한 장** — Before(걸리던 시간) → After(아낀 시간)

Phase A 종료 문구:

```
---
👆 메뉴에서 하나 고르거나 "내 고통점으로 할래"라고 알려주세요.
준비되면 "시작"이라고 입력하면 인터뷰를 시작합니다.
```

여기서 Stop한다. (Phase A에서는 AskUserQuestion·Write를 호출하지 않는다.)

## EXECUTE (Phase B) — 인터뷰 + my-project.md 저장

사용자가 "시작"(또는 메뉴 번호/고통점)을 주면, 아래 순서로 **한 번에 한 주제씩** 인터뷰한다. 4개를 한꺼번에 쏟지 않는다.

### 먼저 — 프로젝트 확정

- 메뉴 번호를 골랐으면 그 프로젝트의 원본 플레이·쓰는 도구를 1줄로 확인해준다.
- "내 고통점"이면 자유서술로 받는다: "매주 반복되는데 시간 잡아먹는 작업 하나만 말해주세요." → 그게 스코핑의 출발점.

### 인터뷰 원칙

- **선택지가 명확한 건 AskUserQuestion, 서술이 필요한 건 자유서술로.**
- 한 주제당 답이 얇으면 **한 번만 더 캐묻는다** (예: "얼마나 자주?"에 "가끔"이라 하면 → "주 몇 회쯤인가요?"). 취조하듯 물고 늘어지지 않는다. 스코핑 1장이 목표지 완벽한 기획서가 아니다.
- **완성 기준은 반드시 "측정 가능"하게 유도한다** (예: "잘 되면"이 아니라 "계정명 입력 → 3분 안에 브리핑"). 실제 딜 1개 적용을 완성 기준에 꼭 넣는다.
- **범위 밖을 반드시 채운다** — 2주는 짧다. 자동 발송·팀 공유용 일반화는 대체로 이번 범위 밖. 참가자가 욕심내면 "그건 2주엔 무리, 범위 밖으로 빼자"고 조율한다.

### 주제 1 — 없애려는 반복 작업

자유서술: "무엇을 / 얼마나 자주 / 지금 회당 몇 분 걸리나요?" 3개를 한 번에 받되, 빠진 게 있으면 한 번만 되묻는다.

### 주제 2 — 완성 기준

자유서술 + 유도: "이게 있으면 '됐다' 싶은 게 뭔가요? 측정 가능하게요." → 최소 2개 체크박스로 정리. 하나는 반드시 "실제 딜 1개에 적용해서 다음 미팅/작업에 썼다".

### 주제 3 — 범위 밖

자유서술 + 유도: "2주 안에 다 하려면 뺄 것도 정해야 해요. 이번엔 안 할 것 하나만 정하면?" 참가자가 막히면 예시: "자동 발송 / 팀 공유용 일반화 / 예외 케이스 전부 처리 — 이 중 하나 빼도 돼요."

### 주제 4 — 쇼케이스 한 장

자유서술: "지금 이 작업에 걸리는 시간(Before)과, 이 프로젝트가 되면 걸릴 시간(After)을 한 줄로." → 2주차 금요일 쇼케이스가 바로 이 before/after 1장이다.

### 파일 작성 — 반드시 실제로 저장한다

4주제가 끝나면:

1. 수집한 내용을 아래 구조(= docs/project-menu.md의 스코핑 1장 템플릿)로 정리한다.
2. **Write 도구로 프로젝트 루트에 `my-project.md`를 저장한다.** (경로: `ai-camp-sales/my-project.md`)
   - 이미 있으면 덮어쓰지 말고 "기존 파일이 있는데 새로 만들까요, 이어붙일까요?"라고 묻는다.
3. 저장 후 내용을 그대로 보여주고 안내한다: "이게 2주차 계약서예요. 수요일 체크인 때 이 기준으로 진행 상황을 봅니다."

작성 템플릿:

```markdown
# 프로젝트: {이름}

## 없애려는 반복 작업
- 무엇을: {작업}
- 얼마나 자주: {주 N회}
- 지금 걸리는 시간: {회당 N분}

## 완성 기준 (이걸 보면 "됐다")
- [ ] {측정 가능한 기준}
- [ ] 실제 딜 1개에 적용해서 다음 미팅/작업에 썼다

## 범위 밖 (이번엔 안 하는 것)
- {제외 항목}

## 쇼케이스 한 장
- Before: {걸리던 시간} → After: {아낄 시간}
```

> 완벽할 필요 없다. **스코핑 1장**이면 된다. 나머지는 2주차 실무에서 채운다.

## QUIZ (Phase B 마무리) — 산출물 확인

```json
AskUserQuestion({
  "questions": [{
    "question": "방금 만든 my-project.md, 지금 상태 어떤가요?",
    "header": "산출물 확인",
    "options": [
      {"label": "범위가 잡혔다 — 저장 완료", "description": "합격. Block 4(수료)로 이동"},
      {"label": "범위가 아직 넓다", "description": "'범위 밖'을 더 채워 2주에 맞게 줄인다"},
      {"label": "저장이 안 됐다", "description": "my-project.md 저장부터 다시 확인"}
    ],
    "multiSelect": false
  }]
})
```

> 정리 한 줄: **범위를 좁힌 스코핑 1장이 2주차의 계약서다.** 넓게 벌리면 못 끝낸다. 좁게 잡고 실제 딜 1개에 쓴다.
