# Block 1: Verify — 까다로운 프로스펙트가 내 결과물을 채점 ⭐

> 오늘의 핵심 블록. 만든 사람이 아닌 "받는 사람"의 눈으로 내 결과물을 본다.
> 이 블록의 메시지: **"내가 만든 걸 내가 채점하면 후하다. 그래서 다른 관점을 빌린다."**
> Phase B에서 Claude는 **Subagent(Task 도구)를 실제로 띄워** 페르소나로 채점한다.

## EXPLAIN (Phase A)

> **Phase A 시작 시 반드시 아래 형태로 출력한다:**
> ```
> 📖 공식 문서: https://code.claude.com/docs/ko/sub-agents
> ```

### Generator ≠ Evaluator

한 사람이 글을 쓰면(Generator), 그 사람은 자기 글을 공정하게 채점(Evaluator)하지 못한다. 내 맥락이 이미 머릿속에 있으니, 빠진 정보도 "당연히 알겠지" 하고 넘긴다. 어제 커스터마이징하면서 **"아쉬운 점 1개"**를 적었던 것 — 그건 사실 빙산의 일각이다. 받는 사람 눈엔 더 많이 보인다.

그래서 오늘은 **채점자를 분리한다.** 만든 나 말고, 이 메일/브리핑을 실제로 받는 **까다로운 프로스펙트**를 Subagent로 띄워서 채점시킨다. Subagent는 내 대화 맥락을 물려받지 않은 **별도 프로세스**라, 내가 "당연히 알겠지" 한 걸 모른 채로 읽는다. 그게 바로 진짜 고객의 상태다.

### 까다로운 프로스펙트 3종

각 페르소나는 내 결과물을 읽고 **반응 3줄 + 점수(10점) + 딱 하나 고칠 것**을 돌려준다.

| 페르소나 | 뭘로 채점하나 | 잘 맞는 결과물 |
|---------|--------------|---------------|
| ① 바쁜 그로스 리드 | 첫 3초에 안 꽂히면 삭제. 나랑 상관있나? | 아웃리치 / 콜드 메일 |
| ② 회의적인 CFO형 | ROI 숫자 없으면 안 믿음. 형용사는 소음 | 업셀 제안 / 가치 브리핑 |
| ③ 이관 피로 담당자 | 전환 리스크 안 다루면 방어적 | 경쟁사 이관 설득 / 배틀카드 |

> 페르소나 프롬프트 전문은 `references/persona-tough-prospect.md`에 있다. 채점할 때 이 파일을 그대로 Subagent에게 넘긴다.

### 오늘 할 것 (한 사이클)

```
내 결과물  →  까다로운 프로스펙트가 채점  →  피드백 읽기  →  딱 1곳만 고침  →  재채점
                                                                          (점수 오르는 걸 체감)
```

## EXECUTE (Phase A에서 안내만, 실제 채점은 Phase B)

참가자에게 안내한다:

**1. 채점받을 결과물 1개를 고른다.** 어제 만든 스킬로 뽑은 **아웃리치 메일**이나 **콜 프렙 브리핑** 하나. (없으면 지금 하나 뽑거나, 운영진 예시를 쓴다.)

**2. 페르소나 1개를 고른다.** 위 표에서 내 결과물에 맞는 것으로.

**3. 아래 한 줄을 복사해 붙여넣고, 그 아래에 내 결과물을 붙인다:**

```
까다로운 프로스펙트로 내 결과물을 채점해줘. 페르소나: 바쁜 그로스 리드

--- 내 결과물 ---
(여기에 아웃리치 메일이나 콜 프렙 브리핑 전문을 붙여넣기)
```

> 페르소나 이름만 바꾸면 된다: "바쁜 그로스 리드" / "회의적인 CFO형" / "이관 피로 담당자".
> 라이브 수업에서는 운영진이 먼저 한 번 시연하고, 참가자가 자기 결과물로 따라 한다.

Phase A 종료 문구:

```
---
👆 위 형식으로 내 결과물을 붙여넣어 주세요.
붙여넣으면 까다로운 프로스펙트가 채점을 시작합니다.
```

여기서 Stop한다. (Phase A에서는 Task·AskUserQuestion을 호출하지 않는다.)

## EXECUTE (Phase B) — 실제 채점 진행 규칙

참가자가 결과물을 붙여넣으면, Claude는 아래대로 **Subagent를 띄워 채점한다.**

### 1. Subagent로 채점 (Generator≠Evaluator를 실제로 지킨다)

- `references/persona-tough-prospect.md`를 읽어 참가자가 고른 페르소나 블록 + 공통 규칙 + 출력 형식을 가져온다.
- **Task 도구로 Subagent를 1개 띄운다.** 지시문 = (그 페르소나 프롬프트 전문) + (참가자가 붙여넣은 결과물). subagent_type은 general-purpose(또는 claude).
- **메인 Claude가 직접 채점하지 않는다.** 반드시 Subagent에게 시킨다 — 그래야 별도 컨텍스트에서 "당연히 알겠지"가 없는 채점이 된다. 이게 이 블록의 핵심이다.
- Subagent가 돌려준 **반응 3줄 + 점수 + 딱 하나 고칠 것**을 그대로 참가자에게 보여준다.

### 2. 피드백 읽고 1곳만 고치기

- 참가자에게 안내한다: "여러 개 고치지 말고, **'딱 하나 고칠 것' 1개만** 반영해보세요. 그 1곳만 고쳐서 다시 가져와 주세요."
- 참가자가 스스로 고치게 둔다. Claude가 대신 다 고쳐주지 않는다(고쳐달라고 하면 그 1곳만 함께 다듬는다).

### 3. 재채점 — 점수 오르는 걸 체감

- 고친 결과물을 받으면 **같은 페르소나로 다시 Subagent를 띄워** 채점한다.
- 점수/반응이 어떻게 달라졌는지 1줄로 짚는다. 이게 오늘의 "체감" 포인트다: **한 곳만 고쳐도 받는 사람 반응이 달라진다.**

> 시간이 남으면 다른 페르소나로 한 번 더 돌려본다(같은 메일도 CFO형과 그로스 리드는 다르게 반응한다). 라이브는 1사이클이면 충분하다.
