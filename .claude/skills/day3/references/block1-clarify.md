# Block 1: Clarify — 모호하게 vs 되묻게 (GIGO 체험)

> 이 블록의 핵심 메시지: **"일 잘 시키는 사람이 AI도 잘 시킨다."**
> 개념을 외우게 하지 않는다. 모호하게 시켜서 실망하고, 되묻게 시켜서 달라지는 걸 **직접 손으로** 겪게 한다.

### GIGO — 쓰레기가 들어가면 쓰레기가 나온다

프로그래밍의 오래된 격언: **Garbage In, Garbage Out.** Claude Code도 똑같다. 아무리 똑똑해도 **모호한 입력을 받으면 모호한 결과**를 낸다.

```
"아웃리치 하나 써줘"
      ↓
Claude: (누구한테? 뭐 팔려고? 톤은? 길이는?)
        → 아무것도 모르니 아무거나 쓴다
        → "아니 그게 아니라..." → 3번 반복
```

머릿속엔 그림이 있는데, Claude에게 전달된 건 단어 몇 개뿐이다. 이 간극이 **모호한 요구의 함정**이다.

### 해결책은 "잘 시키는 것" — Clarify

세일즈는 원래 이걸 잘한다. 신입에게 "고객한테 메일 써"라고 던지지 않는다. **누구에게, 왜, 어떤 톤으로**를 먼저 정한다. AI도 똑같다. 실행 전에 **되묻게** 만들면 결과가 달라진다.

| 모호하게 시킴 | 되묻게 시킴 (Clarify) |
|--------------|----------------------|
| "아웃리치 하나 써줘" | "쓰기 전에 수신자·목적·톤·분량부터 물어봐줘" |
| Claude가 무작위로 하나 고름 | Claude가 4가지를 먼저 확인하고 씀 |
| 결과가 일반적·밋밋 | 결과가 내 상황에 맞음 |

오늘 이후 모든 실습은 이 습관 위에 선다. **"바로 시키기"에서 "묻고 시키기"로.**

## EXECUTE

같은 요청을 **두 번** 시켜보고 결과를 비교한다.

**1단계 — 일부러 모호하게 시킨다.** 아무 맥락 없이 이렇게만:

```
아웃리치 하나 써줘.
```

> 나온 결과를 그대로 읽는다. 고치지 않는다. "이거 그대로 보낼 수 있나?"를 속으로 물어본다. 아마 누구에게 보내는지도, 뭘 파는지도 모르는 일반적인 메일이 나올 것이다.

**2단계 — 이번엔 되묻게 시킨다.** 같은 요청인데, 먼저 물어보라고 한다:

```
아웃리치 하나 쓸 건데, 바로 쓰지 말고
수신자 · 목적 · 톤 · 분량 이 4가지를 먼저 나한테 물어봐줘.
```

> Claude가 AskUserQuestion으로 4가지를 되물을 것이다. 답을 고르고 나서 나온 아웃리치를, 1단계 결과와 나란히 비교한다.

**3단계 — 두 결과를 비교한다.** 무엇이 달라졌는지 눈으로 확인한다. 입력을 조금 더 줬을 뿐인데 결과가 확 달라진다 — 이게 **"잘 시키는 것"의 힘**이다.

비교가 끝나면 "완료" 또는 "다음"이라고 입력해주세요.

## QUIZ

```
