# reply-csm

> AI Camp Day 1~4 학습 효과 측정용 스킬. 같은 고객 이메일에 같은 스킬을 호출하지만, 환경(CLAUDE.md → MCP)이 발전할수록 답변이 달라지는 걸 직접 본다.

---

## 언제 호출되는가

다음 중 하나에 해당하면 호출:

- 사용자가 명시적으로 `/reply-csm` 또는 "reply-csm 스킬 써줘"라고 함
- 사용자가 고객사 이메일을 붙여넣고 "답장 초안 / recap / 회신 좀" 등을 요청
- 사용자가 "리뷰하자 / 처음과 지금 비교해줘 / 내 답변 어떻게 발전했는지" 등을 요청 → **Review 모드**

---

## 두 가지 모드

### A. Reply 모드 (기본)

#### 1단계 — 입력 받기

사용자에게 받은 이메일 본문을 받는다. 형태는 둘 중 하나:
- 메시지 안에 직접 붙여넣은 텍스트
- 파일 경로 (예: `./inbox/customer-email.txt`)

입력이 없으면 짧게 묻는다: "어떤 고객 이메일에 답장하시겠어요? (붙여넣기 또는 파일 경로)"

#### 2단계 — 환경 자동 감지 (모드 결정)

다음 순서로 검사:

1. **CLAUDE.md 존재?** — 현재 작업 디렉토리(cwd) 또는 그 부모 디렉토리에서 `CLAUDE.md` 파일을 찾는다.
   - Bash: `ls CLAUDE.md ../CLAUDE.md 2>/dev/null` 또는 Glob 도구.
2. **MCP 활성?** — 사용자에게 물어보거나, 환경에서 활성 MCP 서버 목록을 확인. Claude Code 환경에서 MCP 서버 리스트를 직접 조회할 수 없으면, 사용자가 환경 정보를 미리 알려주거나 묻는다: "지금 연결된 MCP가 있나요? (Gmail / Notion / Salesforce / 없음)". 답이 "없음"이거나 응답이 없으면 MCP 비활성으로 간주.

판정:

| 조건 | 모드 |
|------|------|
| CLAUDE.md 없음 | **모드 1** |
| CLAUDE.md 있음 + MCP 없음 | **모드 2** |
| CLAUDE.md 있음 + MCP 1개+ | **모드 3** |

> CLAUDE.md의 내용이 비어 있어도 "있으면" 모드 2로 간주 (사용자 결정).

#### 3단계 — 모드별 답장 작성

##### 🧭 모드 1 (CLAUDE.md 없음)

활용할 수 있는 정보: **받은 이메일 본문만**.

작성 가이드:
- 받은 메일에서 명확한 질문/요청을 추출해 답변
- 일반 SaaS 톤 — 형식적이지만 정중함
- 모르는 부분은 "확인 후 답변드리겠습니다" 정도로 안전하게
- 다음 액션은 모호하게 ("회신 부탁드립니다" 수준)
- 절대로 추측해서 구체적인 일정·이름·이력 만들어내지 말 것

##### 🧭 모드 2 (CLAUDE.md 있음)

활용할 수 있는 정보: **받은 이메일 + CLAUDE.md**.

먼저 `CLAUDE.md`를 읽는다. 다음 섹션을 우선적으로 활용:
- 담당 고객 (대표 유형·계약 구조·주요 이슈)
- 톤앤매너 (DO / DON'T)
- 금지어
- 워크플로우 (반복 업무 SOP)

작성 가이드:
- 톤은 CLAUDE.md의 DO를 따른다
- DON'T과 금지어는 절대 사용하지 않는다
- 우리 팀 워크플로우(예: PoC → 결과 같이 보기)를 다음 액션으로 제안
- 그래도 이 고객의 과거 이력은 모르므로, 일반적인 가이드 수준 (날짜·이름 구체화 금지)

##### 🧭 모드 3 (CLAUDE.md + MCP)

활용할 수 있는 정보: **받은 이메일 + CLAUDE.md + 외부 도구(MCP)**.

먼저 CLAUDE.md를 읽고, 활성 MCP를 통해 다음을 조회:
- **Salesforce / CRM MCP**: 이 고객사의 계약 정보, 이전 케이스, 담당자
- **Gmail MCP**: 이 고객과의 과거 메일 스레드
- **Notion MCP**: 내부 문서, 헬프센터, SOP, 미팅 노트
- **Calendar MCP**: 다음 예정된 미팅 (QBR 등)

조회한 정보로 답장 보강:
- 과거 케이스 인용 ("지난 3월 미팅에서 논의된 X와 같은 맥락")
- 구체적 일정·담당자 ("5/8 QBR 때 데이터팀 OO님과 함께")
- 헬프센터 문서 링크
- 내부 SOP 기반 정확한 처리 방식

가이드:
- MCP 조회로 얻은 정보만 인용. 추측·환각 금지.
- MCP 조회 실패 시(권한·검색 결과 없음) → 그 부분은 모드 2 수준으로 fallback
- "모드 3에서 사용한 외부 도구"를 헤더에 명시

#### 4단계 — 출력 형식

```
🧭 모드 N · 활용된 맥락: [리스트, 쉼표 구분]

[recap 답장 본문]

— 리캡 메모리 저장됨 (#N)
```

예시 활용된 맥락 표기:
- 모드 1: `받은 메일만`
- 모드 2: `받은 메일 + CLAUDE.md (담당 고객·톤·금지어)`
- 모드 3: `받은 메일 + CLAUDE.md + Salesforce + Gmail`

#### 5단계 — 리캡 메모리 저장

위치: `.claude/memory/reply-csm/entries.jsonl`

폴더가 없으면 생성한다. 각 호출마다 한 줄(JSONL) 추가:

```json
{"timestamp":"2026-04-25T21:45:00+09:00","mode":2,"context_used":["CLAUDE.md","톤","담당고객"],"input_preview":"BKR CRM&Analytics팀 이에나입니다. $dob 값에...","output_preview":"이에나님, 안녕하세요. 두 건 모두 가능한 방식이라...","entry_number":2}
```

원칙:
- `input_preview` / `output_preview`: 각 200자 이내. 본문 전체 저장 금지 (보안)
- `entry_number`: 기존 entries.jsonl 줄 수 + 1
- 파일이 없으면 생성. 줄 추가는 append만.

푸터의 `(#N)`은 entry_number와 일치.

---

### B. Review 모드

사용자가 "리뷰", "처음과 지금 비교", "내 답변 어떻게 발전" 같은 표현을 쓰면 발동.

#### 1단계 — 메모리 읽기

`.claude/memory/reply-csm/entries.jsonl`을 읽는다. 없거나 entry가 1개뿐이면:
> "아직 비교할 만큼 충분한 호출 이력이 없어요. 모드를 바꿔가며 두 번 이상 호출한 뒤 리뷰해주세요."

#### 2단계 — 첫 entry vs 마지막 entry 비교 출력

```
📊 reply-csm · 처음과 지금 비교

┌─ 첫 호출 (#1) ─────────────────┬─ 마지막 호출 (#N) ─────────────┐
│ 시간: 2026-04-23 14:00          │ 시간: 2026-04-25 21:45           │
│ 모드: 1                         │ 모드: 3                          │
│ 활용 맥락: 받은 메일만           │ 활용 맥락: 메일 + CLAUDE.md +    │
│                                 │           Salesforce + Gmail     │
│                                 │                                  │
│ 결과 미리보기:                   │ 결과 미리보기:                    │
│ "이에나님께, 안녕하세요...       │ "이에나님, 지난 3월 미팅에서     │
│  Import Users 기능을 통한...     │  말씀하셨던 트래킹 정합성...      │
│  추가로 궁금하신 점 있으시면     │  5/8(목) Q2 QBR에 이 어젠다     │
│  회신 부탁드립니다."             │  추가하겠습니다..."              │
└─────────────────────────────────┴─────────────────────────────────┘

✨ 무엇이 달라졌는가
- 정보 풍부도: [한 줄 요약]
- 사실 정확도: [한 줄 요약]
- 톤·말투: [한 줄 요약]
- 다음 액션: [한 줄 요약]
```

분석 1단락은 메모리 entry의 mode·context_used·preview를 비교해서 자동 생성.

---

## 가드레일

- 본문 전체를 메모리에 저장하지 말 것 (200자 preview만)
- MCP 응답이 없으면 fallback 모드로 작동 (절대 환각으로 채우지 말 것)
- 모드 3에서 인용하는 모든 사실은 MCP 조회 결과여야 함. "지난 3월" 같은 표현은 실제 조회된 미팅 날짜일 때만.
- 답장 본문에 "AI가 작성했습니다" 같은 메타 문구 넣지 말 것 — 사용자가 그대로 보낼 수 있는 형태로.
- CLAUDE.md의 금지어는 모드 2·3에서 절대 사용 금지.

---

## 사용 시나리오 (캠프 흐름)

| 시점 | 환경 | 모드 | 메모리 entry |
|------|------|------|--------------|
| Day 1 시작 | CLAUDE.md 없음 | 1 | #1 |
| Day 1 후반 | CLAUDE.md 작성 후 | 2 | #2 |
| Day 2 후반 | CLAUDE.md + MCP 1개+ | 3 | #3 |
| Day 4 회고 | "리뷰하자" | (Review) | #1 ↔ #3 비교 |

같은 고객 이메일을 다시 가져와 호출하면, 환경 변화에 따라 결과가 어떻게 달라지는지 자기 눈으로 본다.
