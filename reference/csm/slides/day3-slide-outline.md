## B1 · Deep Interview + Clarify (30분 · 10장) · S09–S18

### S09 · 섹션 오프너
- template: section divider
- eyebrow: `Block 1`
- headline: **Deep Interview 스킬 실습 — 문제를 구체 스펙으로**
- sub: 요청을 바로 실행시키지 않는다. 먼저 묻고 다듬는다.

### S10 · Clarify 개념 (STEP 1/2)
- template: concept card (정의 + 왜)
- headline: **Clarify = "원하는 것"을 말로 만드는 단계**
- 정의: AI가 바로 실행하기 전에 **"정확히 무엇을 만들까"를 되묻는** 절차.
- 왜: 머릿속 이미지는 대체로 두 개 정답 사이를 오간다. 묻지 않으면 AI가 무작위로 하나를 고른다.

### S11 · Clarify 개념 (STEP 2/2)
- template: 3-row before/after
- headline: **같은 요청, Clarify 전 vs 후**
- rows:
  - `"이탈 위험 리포트 써줘"` → Clarify 후 "독자·핵심 질문 3개·결론 먼저"까지 구체화
  - `"리캡 메일 초안 줘"` → "상대 · 회의 유형 · 결정사항 강조 여부"까지 지정
  - `"문의 답변 드래프트"` → "톤 · 회피 표현 · CTA 포함 여부"까지 지정
- closing: Clarify 없는 스킬은 "빠르게 틀린 답"을 만든다.

### S12 · Ouroboros 비유 (한 줄)
- template: analogy one-liner (terra accent)
- eyebrow: `비유`
- headline: **오늘 만드는 스킬 = 내일의 나를 위한 단축키**
- 본문: 스킬이 내 손을 대신 잡고 Clarify까지 처리해준다. 쓸수록 더 잘 쓰인다.

### S13 · Deep Interview 스킬 소개 (STEP 1/2)
- template: concept card (정의 + 왜 중요 + CSM 예시)
- headline: **Deep Interview = Clarify를 구조로 박은 스킬**
- 정의: 문제를 입력하면 AI가 **"누구를 위해, 어떤 포맷, 성공 기준은 무엇"**을 단계적으로 묻고 스펙 문서로 정리한다.
- 왜 중요: 즉흥 대화로 Clarify 할 때보다 빠짐이 적다. 스펙을 **재사용 가능한 문서**로 남긴다.
- CSM 예시: "이 고객 이탈 위험 리포트" 주제 → Deep Interview가 독자·포맷·결론 구조를 질문지 형태로 묶어줌.

### S14 · Deep Interview 스킬 소개 (STEP 2/2)
- template: 4-step flow
- headline: **Deep Interview 실행 흐름**
- 4 steps:
  1. 문제 1줄 입력 — "나 이런 걸 만들고 싶어"
  2. Deep Interview 스킬 호출 — 구조화 질문 시작
  3. 독자 · 입력 · 출력 · 성공 기준 응답
  4. 스펙 문서 산출 — 다음 블록 스킬 커스터마이징에서 재료
- note: 짝꿍이 옆에서 "이 질문에 왜 그렇게 답했나" 되묻기 역할.

### S15 · 실습 · 문제 고르기 (실습 1/4)
- template: mission card
- eyebrow: `실습 · 5분`
- headline: **Day 2에서 느낀 답답함 중 1개 고르기**
- body:
  - "반복될 것" — 월 1회 이상 생길 법한 일
  - "내가 손으로 하고 있음" — 지금 수동 작업이 있는 단계
  - "지금 설명하면 AI가 흉내는 내지만 내 것 같지 않음"
- foot: 못 고르면 운영진 샘플 3개(리캡 메일 / 문의 답변 / 체크업 미팅 자료) 중 선택.

### S16 · 실습 · Deep Interview 호출 (실습 2/4)
- template: command guide (4-step)
- eyebrow: `실습 · 10분`
- headline: **Deep Interview 스킬 호출 순서**
- steps:
  - `Claude Code 세션 진입` · 새 폴더 OK
  - `문제 1줄 입력` · "고객 X 리캡 메일 스킬을 만들고 싶어"
  - `Deep Interview 스킬 실행` · 자동 질문 시작
  - `5~8개 질문에 답하기` · 모르는 건 "모른다"고 답해도 OK
- note: 첫 시도는 대답이 짧아도 된다. 스펙 문서를 받아내는 게 목표.

### S17 · 실습 · 스펙 받기 (실습 3/4)
- template: output inspection (3-point)
- eyebrow: `실습 · 10분`
- headline: **받은 스펙을 읽고 3가지만 체크**
- 3 points:
  - **독자** — "누가 읽는가"가 한 문장으로 명시됐나
  - **입력/출력** — 재료와 결과 형태가 명시됐나
  - **성공 기준** — "이 스킬이 잘 됐다" 판정 기준이 있는가
- foot: 3개 다 맞으면 다음 블록 준비 완료. 허전하면 그 항목만 추가 질문.

### S18 · 실습 · 짝꿍 점검 (실습 4/4) · 정리
- template: pair exercise
- eyebrow: `짝꿍 · 5분`
- headline: **스펙 한 번 읽어주기**
- 질문 2개:
  1. "이 스펙만 보고 스킬이 어떤 결과를 낼지 그려지나?"
  2. "빠진 항목 1개 짚어주기"
- closing: 완벽 X. 다음 블록에서 운영진 베이스에 이 스펙을 얹으면 초안이 된다.

## B2 · 스킬 커스터마이징 0.5→1 (60분 · 14장) · S19–S32

### S19 · 섹션 오프너
- template: section divider (terra full bleed)
- eyebrow: `Block 2 · 60분`
- headline: **스킬 커스터마이징 — 운영진 베이스를 내 것으로**
- sub: 0에서 시작 X. 운영진 베이스 스킬을 내 맥락으로 수정해 첫 초안을 만든다.

### S20 · 운영진 베이스 템플릿 소개 (STEP 1/2)
- template: 3-card base template
- headline: **오늘 쓸 베이스 스킬 3종**
- intro: 운영진이 사전 제작한 CSM 핵심 3종. 수정만 해도 동작하는 수준.
- 3 cards:
  - **리캡 메일 스킬** — 미팅 노트 입력 → 리캡 메일 초안. 톤·CTA 슬롯.
  - **문의 답변 스킬** — 문의 본문 입력 → 답변 초안. 회피 표현·레퍼런스 링크 슬롯.
  - **체크업 미팅 자료 스킬** — 고객 정보 입력 → 아젠다 + 지표 요약. 지표 소스 슬롯.
- foot: 1개만 골라서 시작. 끝나면 나머지는 2주 프로젝트에서.

### S21 · 운영진 베이스 템플릿 소개 (STEP 2/2)
- template: file structure card
- headline: **스킬 하나는 어떻게 생겼나**
- 본문: 베이스 스킬은 **세 블록**으로 구성.
  - `SKILL.md` — 스킬의 정체성 · 호출 조건 · 대표 예시
  - `templates/` — 출력 포맷 (예: 리캡 메일 본문 틀)
  - `rules/` — 톤 · 금지어 · 필수 섹션
- closing: 이 3개 중 **2개는 그대로 두고 1개만 고치기**부터 시작.

### S22 · Day 2 스킬과 결합 (STEP 1/2)
- template: concept card
- headline: **어제 연결한 MCP가 이 스킬에 어떻게 쓰이나**
- 본문: 베이스 스킬은 `SKILL.md` 상단에 "이 스킬은 Notion MCP · Gmail MCP를 사용한다"로 선언. 실행 시 AI가 그 도구를 자동 호출.
- 예시: 리캡 메일 스킬 → Notion MCP에서 지난 미팅 노트 읽고 · Gmail MCP로 지난 메일 톤 참고.

### S23 · Day 2 스킬과 결합 (STEP 2/2)
- template: 3-line diagram (CLAUDE.md → 스킬 → MCP)
- headline: **세 층이 협력한다**
- 3 lines:
  - CLAUDE.md — 내 기본값 (담당 고객 · 톤)
  - 스킬 — 반복 작업의 절차
  - MCP — 실시간 맥락 (그 고객 지난 미팅 · 메일 스레드)
- closing: 오늘 커스터마이징은 "스킬 층"만 손대면 된다. 나머지 두 층은 이미 쌓여있음.

### S24 · 수정 3요소 · 담당 고객 (요소 1/3)
- template: before/after card
- headline: **수정 포인트 1 — 담당 고객**
- intro: 베이스 스킬은 "일반 CSM" 기본값. 내 담당 고객군으로 치환.
- before (cream): "고객사 정보는 SFDC에서 가져온다" (일반)
- after (terra): "고객사 A·B·C 우선순위 순서로 SFDC 조회, 없으면 Notion fallback" (내 맥락)
- foot: CLAUDE.md에 이미 담당 고객이 있다면 "CLAUDE.md 참고"로 한 줄만 적어도 됨.

### S25 · 수정 3요소 · 톤 (요소 2/3)
- template: before/after card
- headline: **수정 포인트 2 — 톤**
- intro: 기본 템플릿 톤은 보수적 중립. 내 고객군 따라 조정.
- before (cream): "정중하고 간결한 비즈니스 톤"
- after (terra):
  - 엔터프라이즈 고객 중심이면 — "임원 회람 가능성 고려, 결론 먼저, 한 문단 3줄"
  - 스타트업 고객 중심이면 — "실무자 직접 소통, 기술 용어 OK, 짧은 단락"
- foot: 톤은 rules/에 한 문단으로. 예시 문장 1-2개 같이 박기.

### S26 · 수정 3요소 · 키워드 (요소 3/3)
- template: before/after card
- headline: **수정 포인트 3 — 키워드 (회피 · 필수)**
- intro: "쓰면 안 되는 말"과 "반드시 들어가야 할 말"이 CSM에게 가장 중요.
- before (cream): "금지어 없음 / 필수 키워드 없음"
- after (terra):
  - 금지어: "죄송합니다" 단독 사용 · "확인 중입니다" 마무리 · 내부 프로젝트 코드네임
  - 필수: 다음 액션 날짜 · 담당자 이름 · 관련 문서 링크 1개
- closing: 이 3요소만 맞추면 베이스 스킬은 **내 것의 70%**까지 간다.

### S34 · Subagent 개념 2분
- template: concept card (정의 + 왜)
- headline: **Subagent = 한 번에 여러 에이전트 병렬 호출**

- 왜: 순차로 하면 3번 호출 = 3배 시간. 병렬이면 거의 1배.
- 일회성: 대화 맥락을 이어받지 않는 **일회용 작업자**. 맥락 오염 X.

### S35 · Subagent vs 일반 호출

  - → 메일 톤 분석 (30초)
  - → 리캡 메일 작성 (30초)
  - 총 90초
- right (terra) · Subagent 병렬:
  - 히스토리 요약 + 톤 분석을 **동시에** 돌림 (30초)
  - → 리캡 메일 작성 (30초)
  - 총 60초
- foot: 차이는 40초 아니라 **"내가 기다리는 시간이 반"**.

  4. 수정안 1개만 반영 — 전부 X · 가장 영향 큰 1개
