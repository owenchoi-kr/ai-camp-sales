# battlecard

> 경쟁 MMP 계정을 이관 설득하는 배틀카드를 만든다. 사실 기반 비교만 하고, 경쟁사 비방은 절대 생성하지 않는다.

## 0. Codex 실행 노트

- description 매칭으로 자동 호출되거나 `$battlecard` 로 명시 호출.
- 컨텍스트 파일은 **AGENTS.md**.
- 전환 사례 소스는 `config.toml` 의 Slack/Notion MCP. 없으면 사례 슬롯을 비워둔다.

## 1. 입력 받기

- **계정명** (필수)
- **현재 사용 MMP(경쟁사)** — 예: AppsFlyer / Adjust 등
- **계약 갱신 시기** — 갱신 타이밍이 곧 이관 창구다

없으면 짧게 묻는다.

## 2. 환경 자동 감지 (모드 결정)

1. **AGENTS.md 있는가?** — `ls AGENTS.md ../AGENTS.md 2>/dev/null`
2. **전환 사례 소스가 있는가?** — Slack MCP(과거 이관 논의 검색) 또는 Notion MCP(사례 페이지)가 연결돼 있는가?

| 조건 | 모드 |
|------|------|
| AGENTS.md 없음 | 모드 1 |
| AGENTS.md 있음 + 사례 소스 없음 | 모드 2 |
| AGENTS.md 있음 + 사례 소스 1개+ | 모드 3 |

## 3. 모드별 동작

**모드 1 (AGENTS.md 없음)** — 입력한 계정·경쟁사·갱신 시기만으로 일반적인 이관 설득 프레임을 작성한다. 고객 pain은 해당 버티컬의 일반적 과제 수준으로만 다룬다.

**모드 2 (AGENTS.md 있음)** — `AGENTS.md`를 읽어 {{주력_버티컬}}에 맞춘 pain 포인트로 설득 논리를 좁힌다. 예상 반론은 {{자주_만나는_반론}}을 우선 배치한다. {{금지_표현}}은 결과물에 절대 넣지 않는다.

**모드 3 (AGENTS.md + 사례 소스)** — 위에 더해, Slack/Notion에서 실제 이관 성공 사례를 검색해 "유사 버티컬 전환 사례" 슬롯을 근거와 함께 채운다. 조회 실패 시 그 슬롯은 "사례 미확보 — 채워 넣을 것"으로 비워둔다.

## 4. 출력 형식

```
🧭 모드 N · 활용된 맥락: [AGENTS.md / Slack 사례 / ...]

# 배틀카드 — {계정명} · {경쟁사} → 우리 제품
갱신 시기: {갱신 시기}

## 전환 설득 포인트 (고객 pain 기반)
- {{주력_버티컬}}에서 이 고객이 겪을 pain → 우리가 푸는 방식 3개

## 예상 반론 & 대응
| 반론 | 대응 (사실 기반) |
|------|------------------|
| "이관 리스크가 크다" | ... |
| "기존 데이터가 유실된다" | ... |
| {{자주_만나는_반론}} | ... |

## 유사 버티컬 전환 사례
- [모드 3에서 조회한 사례 / 없으면 "미확보"]
```

## 가드레일

- **경쟁사 비방 표현 생성 금지.** "쓰레기", "형편없는" 같은 감정적 폄하 대신, 검증 가능한 사실 비교(기능 유무·지원 범위·정합성)만 쓴다.
- {{금지_표현}}에 넣은 표현은 어떤 경우에도 출력하지 않는다.
- **모든 사실 주장에는 출처(URL) 또는 `[가설]` 라벨을 단다.** 라벨 없는 단정 금지. 경쟁사 스펙은 "확인된 범위"로만 서술한다.
- **고객 사례는 1차 출처(공식 사이트·보도자료)로 확인된 것만 인용한다.** 매칭되는 사례가 없으면 추측으로 채우지 말고 "공개 사례 없음"으로 표기한다. (거짓 레퍼런스가 실제 영업 사고로 이어진 사례가 있다.)
